# 發票辨識結算帳冊 — 架構與功能說明（新手導向）

這份文件用白話說明整個系統「做了什麼、為什麼這樣做、怎麼做到」，方便日後維護或交接。

---

## 一、三層架構（最重要的觀念）

```
[使用者瀏覽器]  index.html              ← 前端：畫面、按鈕、表格、圖表
      │  （fetch 發 HTTP 請求）
      ▼
[Cloudflare 邊緣] functions/api/*.js     ← 後端：保管金鑰、呼叫 AI、讀寫資料
      │
      ▼
[外部服務]  Gemini API / 本地 Ollama / Cloudflare KV
```

**為什麼分前後端？**
API 金鑰、密碼**絕不能放前端**——前端程式任何人按 F12 都看得到。所以金鑰只存在後端（Cloudflare Secret / 環境變數），前端只能「請後端代勞」。這是整個設計的安全核心。

| 檔案 | 角色 |
|---|---|
| `index.html` | 前端，跑在瀏覽器，負責所有畫面與互動 |
| `functions/api/recognize.js` | 後端：辨識發票（呼叫 AI） |
| `functions/api/units.js` | 後端：讀寫「使用單位」清單 |
| `wrangler.toml` | 專案設定 + 綁定（例如 KV 資料庫） |

部署平台是 **Cloudflare Pages**（不是 Workers）。`git push` 到 GitHub 後，Cloudflare 會自動重新部署。

## 二、功能與設計理由

### 1. Gemini 超載時的「倒數重試」
- **問題**：Gemini 忙碌時回 HTTP `503`，原本直接報錯、那張發票就廢了。
- **做法**：後端偵測到 `429`（請求太多）/`503`（服務忙）時，回一個「可重試」訊號與建議等待秒數；前端顯示倒數，時間到自動重新辨識。
- **觀念**：HTTP 狀態碼是判斷「該不該重試」的標準依據。`4xx` 多半是請求方的錯，`5xx` 多半是對方伺服器的錯，其中 `429/503` 通常稍等可重試。

### 2. Gemini 掛掉就改用「本地端 Ollama」（fallback 後備機制）
- **問題**：完全依賴雲端 Gemini，一出問題就停擺。
- **做法**：後端先試 Gemini，**一失敗就改打本地用 RTX 4090 跑的 Qwen2.5-VL**（經 Cloudflare Tunnel 對外）。回應帶 `engine` 欄位（`gemini` / `local`），前端顯示「⚙ 本地端讀取」標籤。
- **切換條件**：後端 `const hasLocal = env.AI_BASE_URL && env.AI_API_KEY`——這兩個 Secret 都有值才會啟用本地後備。
- **觀念**：**fallback** = 主要服務失敗時自動切備援，提高可用性。

### 3. 可編輯的「使用單位」清單（用到資料庫 KV）
- **問題**：要幫每張發票標單位（餐飲部、客房部…），清單要存起來下次還能用。
- **做法**：
  - 用 **Cloudflare KV**（簡單的「鍵—值」資料庫）存整份清單。
  - `functions/api/units.js`：`GET` 讀清單、`POST` 存清單，沿用 `APP_PASSWORD` 保護。
  - 前端右下角懸浮面板可新增/刪除/勾選顯示，**改動就自動存後端**（debounce 0.6 秒）。
  - **兩層單位**：發票表頭選「母單位」；每個品項可選「覆蓋單位」，沒選就跟隨母單位。
    `effUnit(inv,it) = 品項覆蓋 → 否則母單位 → 否則「未指定」`。
- **觀念**：
  - **持久化**：把資料存資料庫，重開頁面還在（記憶體一刷新就沒）。
  - **debounce（防抖）**：連續改動時等你停手才送一次，避免狂打後端。

### 4. 彈性分析彈窗 + 圖表
- **問題**：固定樞紐表無法自由切換維度或篩選。
- **做法**：
  - 工具列「📊 彈性分析」開**全螢幕彈窗**。
  - 控制列：維度（單位/廠商）、篩選（廠商/單位）、細分到品項、圖表（圓餅/長條）。
  - 用 **Chart.js**（CDN 載入的圖表庫）畫圖。
- **觀念**：
  - **以品項層級計算**：單位掛在品項上，所以一個品項一個品項加總，金額才不重複。
  - **重畫前先 `chart.destroy()`**：否則新圖會疊在舊圖上——用圖表庫常見的坑。
  - **CDN**：別人代管的函式庫網址，`<script src>` 一行就能用。

## 三、Cloudflare 設定（Secret 與綁定）

這些都在 Cloudflare 後台設定，**不寫進程式碼**。

| 名稱 | 類型 | 用途 |
|---|---|---|
| `APP_PASSWORD` | Secret | 登入密碼（必填） |
| `GEMINI_API_KEY` | Secret | Gemini 金鑰 |
| `AI_BASE_URL` | Secret | 本地 Ollama 的 OpenAI 相容端點（結尾 `/v1`） |
| `AI_API_KEY` | Secret | Ollama 不驗證，填 `ollama` 即可 |
| `AI_MODEL` | Secret | 本機模型名，如 `qwen2.5vl:7b` |
| `UNITS_KV` | KV 綁定 | 「使用單位」清單的資料庫 |

> 因為本專案有 `wrangler.toml`，**綁定（如 KV）要寫在 `wrangler.toml`**，dashboard 的綁定按鈕會被鎖住。KV 綁定範例：
> ```toml
> [[kv_namespaces]]
> binding = "UNITS_KV"
> id = "<namespace 的 ID>"
> ```

**本地 Ollama 前置**：`http://localhost:11434` 顯示 "Ollama is running"，且 tunnel 要這樣開（少 `--http-host-header` 會被回 403）：
```
cloudflared tunnel --url http://localhost:11434 --http-host-header localhost:11434
```

## 四、可複用的開發套路

1. **新功能碰到金鑰/資料** → 走「前端 fetch → 後端 function → 外部服務」，不要在前端直接接外部 API。
2. **要存資料** → KV（簡單清單）或 D1（關聯式）；先建 → `wrangler.toml` 綁定 → 程式用 `env.XXX` 取用。
3. **改完程式** → `git commit` → `git push` → Cloudflare 自動部署；改 Secret/綁定有時要「Retry deployment」。
4. **判斷錯誤要不要重試** → 看 HTTP 狀態碼（429/503 可重試）。

## 五、本機 / 部署速查

- 改前端：直接編輯 `index.html`。
- 改後端：編輯 `functions/api/*.js`。
- 發佈：`git add -A && git commit -m "..." && git push`，Cloudflare 自動部署。
- 工作檔：畫面可「儲存工作檔 (.json)」保存當前資料，之後「載入」還原。
